En este capítulo se describe la metodología empleada para el desarrollo de la plataforma web,
desde la fase de análisis de requerimientos hasta la unificación de todas las partes del proyecto.

La metodología aplicada se fundamentó en un enfoque estructurado de cinco fases secuenciales e
interrelacionadas, que permitieron una gestión eficiente del proyecto y una integración fluida de
los distintos productos finales de cada fase.

\section{Fase 1: Recolección de Requerimientos}
La primera fase se centró en la extracción de requerimientos técnicos y funcionales a partir del
análisis de los diseños de Interfaz de Usuario previamente elaborador por Andrés Rodríguez en
\cite{Rodríguez_Dis_2023}.

Esta fase tiene como objetivo principal identificar y documentar las reglas de negocio, flujos de
información, roles, restricciones y validaciones necesarias para que la plataforma opere de manera correcta.

\subsection{Definición de módulos funcionales}
Con el fin de identificar acciones, validaciones y flujos de navegación se realizó un análisis detallado
de cada una de las páginas, componentes y formularios presentes en los diseños de interfaz.
Este análisis permitió extraer los módulos funcionales que permitirán un soporte alineado y coherente con 
el fin de la plataforma.
Los módulos funcionales identificados fueron:
\begin{itemize}
    \item \textbf{Módulo de Autenticación:} Este módulo se encarga del registro, inicio de sesión, recuperación
            de contraseña, \textit{logout} y gestión de sesiones de los usuarios. Además, la asignación automática
            de roles a los usuarios.
    \item \textbf{Módulo de Usuarios:} Este módulo se encarga de la actualización de información, cambio de contraseña,
            aprobación de cuentas para usuarios con roles específicos, vista de perfiles. 
    \item \textbf{Módulo de Proyectos:} Este módulo se encarga de la creación, edición, eliminación y visualización de
            proyectos, así como la gestión de sus estados y la aprobación de proyectos por parte de usuarios con roles específicos.
    \item \textbf{Módulo de Postulaciones:} Este módulo se encarga de la gestión de postulaciones a proyectos, incluyendo
            la postulación, revisión y aprobación de postulaciones por parte de usuarios con roles específicos. Además, de
            permitir la visualización de historial de postulaciones.
    \item \textbf{Módulo de Horas:} Este módulo se encarga de la gestión de horas dedicadas a proyectos, incluyendo el registro,
            revisión y aprobación de horas por parte de usuarios con roles específicos. Además, de permitir la visualización de
            historial de horas.
    \item \textbf{Módulo de Archivos:} Este módulo se encarga de la gestión de archivos relacionados con usuarios, proyectos y
            horas, incluyendo la creación y eliminación de archivos, así como la visualización de archivos relacionados a cada entidad.
\end{itemize}

\subsection{Análisis de formularios y mapeo de datos}
Alineado con los módulos funcionales, cada formulario presente en los diseños de interfaz fue desglosado para identificar los
campos de entrada, sus tipos de datos y validaciones necesarias. Permitiendo establecer un mapeo claro entre los datos requeridos
por la plataforma para cada acción posible.

\subsection{Definición de roles y permisos}
A partir de los diseños de interfaz fue sencillo identificar los distintos roles de usuario y sus respectivas acciones permitidas.
De esta manera, se definieron los roles y permisos por usuario, resultando en la siguiente clasificación:
\begin{itemize}
    \item \textbf{Todos los usuarios:}
    \begin{itemize}
        \item Login y registro.
        \item Verificación de correo electrónico.
        \item Restablecimiento de contraseña.
        \item Visualización de proyectos.
        \item Visualización de perfiles de usuario.
        \item Visualización de archivos relacionados a cada entidad.
        \item Visualización de las motivaciones.
        \item Actualización de información personal y cambio de contraseña.
    \end{itemize}
    \item \textbf{Estudiantes:}
    \begin{itemize}
        \item Postulación a proyectos.
        \item Participación en proyectos.
        \item Abandonar proyectos.
        \item Registro de horas trabajadas con evidencias.
        \item Visualización de horas aprobadas en proyectos.
        \item Visualización de historial de postulaciones y horas.
    \end{itemize}
    \item \textbf{Organizaciones externas:}
    \begin{itemize}
        \item Creación, edición y eliminación de proyectos con archivos (recursos) ligados.
        \item Iniciar, finalizar y cancelar proyectos propios.
        \item Visualización de postulaciones a proyectos propios.
        \item Aprobación de postulaciones de proyectos propios.
        \item Visualización de participantes de proyectos propios.
        \item Eliminar participantes de proyectos propios.
        \item Aprobación de horas trabajadas por estudiantes en proyectos propios.
    \end{itemize}
    \item \textbf{Directores:}
    \begin{itemize}
        \item Lo mismo que organizaciones externas.
        \item Aprobación de proyectos enviados por organizaciones externas.
        \item Aprobación de cuentas de organizaciones externas.
    \end{itemize}
\end{itemize}

\subsubsection{Estandarización de estados lógicos}
Con el fin de estandarizar estados lógicos, evitando inconsistencias, se definieron tipos
enumerados (ENUMs) para los roles de los usuarios y estados de proyectos, postulaciones y horas.

\subsubsection{Entidades}
\begin{itemize}
    \item \textbf{Campuses:} Almacena los distintos campus universitarios disponibles.
    \item \textbf{Faculties:} Almacena la información de las facultades académicas.
    \item \textbf{Careers:} Almacena las carreras académicas disponibles.
    \item \textbf{Users:} Almacena la información de los usuarios registrados, principalmente incluyendo información de
            autenticación y rol.
    \item \textbf{Students:} Almacena la información académica y personal de los estudiantes.
    \item \textbf{Directors:} Almacena la información académica de los directores académicos.
    \item \textbf{DirectorFaculties:} Tabla intermedia que permite vincular directores y facultades.
    \item \textbf{DirectorCareers:} Tabla intermedia que permite vincular directores y carreras.
    \item \textbf{Organizations:} Almacena la información de las organizaciones externas.
    \item \textbf{RefreshTokens:} Almacena los tokens de autenticación persistente utilizados para renovar sesiones de usuario.
    \item \textbf{PasswordResetTokens:} Almacena tokens temporales de un solo uso para el proceso de recuperación de contraseña.
    \item \textbf{EmailVerificationTokens:} Almacena los tokens utilizados para verificar la dirección de correo electrónico
            de los usuarios.
    \item \textbf{Projects:} Almacena los proyectos disponibles en la plataforma. Incluyendo información como título, descripción,
            fechas, estado, requisitos, entre otros.
    \item \textbf{ProjectCareers:} Tabla intermedia que permite vincular proyectos con carreras.
    \item \textbf{Tags:} Almacena las etiquetas disponibles para categorizar proyectos.
    \item \textbf{ProjectTags:} Tabla intermedia que permite vincular etiquetas con proyectos.
    \item \textbf{Applications:} Almacena las postulaciones de los estudiantes a proyectos específicos, incluyendo el estado de
            la aplicación.
    \item \textbf{Hours:} Almacena las horas realizadas por los estudiantes en los proyectos, incluyendo el tipo de hora,
            cantidad y estado de aprobación.
    \item \textbf{Files:} Almacena los archivos asociados al sistema, como fotografías de perfil, recursos de proyectos o
            evidencias de horas.
\end{itemize}

\subsubsection{Relaciones}
\begin{itemize}
    \item Un campus puede tener múltiples facultades, pero cada facultad pertenece a un único campus.
    \item Una facultad puede tener varias carreras, pero cada carrera pertenece a una sola facultad.
    \item Un usuario puede estar asociado a un único estudiante, director u organización.
    \item Un director puede tener varias facultades.
    \item Un director puede tener varias carreras.
    \item Una organización puede crear múltiples proyectos, pero cada proyecto pertenece a una única organización (cuando aplica).
    \item Un director puede crear múltiples proyectos, pero cada proyecto pertenece a un único director (cuando aplica).
    \item Un proyecto puede estar disponible para varias carreras y una carrera puede participar en múltiples proyectos.
    \item Un proyecto puede recibir múltiples postulaciones, pero cada postulación corresponde a un único proyecto.
    \item Un estudiante puede postularse a múltiples proyectos, pero cada postulación pertenece a un único estudiante.
    \item Un estudiante puede registrar múltiples transacciones de horas en distintos proyectos.
    \item Un proyecto puede tener múltiples registros de horas asociados.
    \item Un usuario puede subir múltiples archivos.
    \item Un proyecto puede tener múltiples \textit{tags} asociados, y una \textit{tag} puede estar asociada a múltiples proyectos.
    \item Un proyecto puede tener múltiples archivos asociados como recursos.
    \item Un registro de horas puede tener múltiples archivos asociados como evidencias.
    \item Un usuario puede tener múltiples tokens de autenticación, verificación o recuperación asociados.
\end{itemize}

\subsection{Definición de \textit{Endpoints}}
A partir de los módulos funcionales y el análisis de formularios se estructuró la arquitectura \textit{RESTful} de la API.
Definiendo contratos de comunicación claros, incluyendo rutas, verbos HTTP, parámetros de entrada, formatos de respuesta
y códigos de estado.

\subsubsection{Autenticación}
Ruta base: \texttt{/auth}. Este conjunto de \textit{endpoints} se encarga de la autenticación, recuperación de contraseña,
verificación de correo electrónico, gestión de sesiones y tokens de autenticación.
\begin{itemize}
    \item \texttt{POST /auth/register}: Registro de usuarios.
    \item \texttt{POST /auth/login}: Inicio de sesión.
    \item \texttt{POST /auth/refresh}: Renovación de tokens de autenticación.
    \item \texttt{POST /auth/verify-email}: Verificación de correo electrónico.
    \item \texttt{POST /auth/forgot-password}: Solicitud de restablecimiento de contraseña.
    \item \texttt{POST /auth/reset-password}: Confirmación de restablecimiento de contraseña con token.
    \item \texttt{POST /auth/logout}: Cierre de una sesión específica.
    \item \texttt{POST /auth/logout-all}: Cierre de todas las sesiones activas.
    \item \texttt{GET /auth/sessions}: Listado de las sesiones activas.
\end{itemize}

\subsubsection{Usuarios}
Ruta base: \texttt{/users}. Este conjunto de \textit{endpoints} gestiona la información de los usuarios registrados,
incluyendo consulta de perfiles, actualización de datos, cambio de contraseña y eliminación de cuentas.
\begin{itemize}
    \item \texttt{GET /users/}: Listar estudiantes.
    \item \texttt{GET /users/me}: Obtener perfil propio.
    \item \texttt{GET /users/:id/profile}: Obtener perfil público de un usuario.
    \item \texttt{GET /users/:userId}: Obtener usuario por ID (administrador).
    \item \texttt{PUT /users/:id}: Actualizar información de usuario.
    \item \texttt{PUT /users/:id/change-password}: Cambiar contraseña.
    \item \texttt{DELETE /users/:id}: Eliminar usuario.
    \item \texttt{PUT /users/:id/profile-picture}: Actualizar fotografía de perfil.
    \item \texttt{GET /pending-organizations}: Obtener lista de organizaciones pendientes de aprobación.
    \item \texttt{PATCH /users/:id/approve}: Aprobar o rechazar cuenta de organización.
\end{itemize}

\subsubsection{Proyectos}
Ruta base: \texttt{/projects}. Este conjunto de \textit{endpoints} gestiona la creación, consulta, actualización,
aprobación y eliminación de proyectos.
\begin{itemize}
    \item \texttt{GET /projects}: Listar proyectos con filtros.
    \item \texttt{GET /projects/my-projects}: Listar proyectos propios.
    \item \texttt{GET /projects/pending-approvals}: Listar proyectos pendientes de aprobación o rechazo.
    \item \texttt{GET /projects/:projectId}: Obtener proyecto por ID.
    \item \texttt{POST /projects}: Crear proyecto.
    \item \texttt{PUT /projects/:projectId}: Actualizar proyecto.
    \item \texttt{POST /projects/:projectId/approve}: Aprobar o rechazar proyecto.
    \item \texttt{PATCH /projects/:projectId/status}: Cambiar estado del proyecto.
    \item \texttt{DELETE /projects/:projectId}: Eliminar proyecto.
\end{itemize}

\subsubsection{Aplicaciones}
Ruta base: \texttt{/applications}. Este conjunto de \textit{endpoints} gestiona las postulaciones de estudiantes
a proyectos y la administración de participantes.
\begin{itemize}
    \item \texttt{POST /applications}: Aplicar a un proyecto.
    \item \texttt{GET /applications/completed}: Obtener historial de proyectos finalizados.
    \item \texttt{GET /applications/project/:projectId}: Obtener aplicaciones de un proyecto.
    \item \texttt{GET /applications/project/:projectId/participants}: Obtener participantes aceptados de un proyecto.
    \item \texttt{PATCH /applications/:applicationId/status}: Aprobar o rechazar aplicación.
    \item \texttt{PATCH /applications/project/:projectId/leave}: Abandonar proyecto.
    \item \texttt{PATCH /applications/project/:projectId/participant/:studentId}: Eliminar participante de un proyecto.
\end{itemize}

\subsubsection{Horas}
Ruta base: \texttt{/hours}. Este conjunto de \textit{endpoints} gestiona el registro, consulta, aprobación
y reenvío de horas asociadas a proyectos.
\begin{itemize}
    \item \texttt{POST /hours}: Registrar horas con evidencias.
    \item \texttt{GET /hours/total}: Obtener total de horas aprobadas.
    \item \texttt{GET /hours/:hourId}: Obtener detalles de un registro de horas.
    \item \texttt{GET /hours/project/:projectId/pending}: Obtener horas pendientes de aprobación en un proyecto.
    \item \texttt{PATCH /hours/:hourId/status}: Aprobar o rechazar horas.
    \item \texttt{POST /hours/:hourId/evidences}: Agregar evidencias a registro.
    \item \texttt{DELETE /hours/:hourId/evidences/:fileId}: Eliminar evidencia específica.
    \item \texttt{PATCH /hours/:hourId/resubmit}: Reenviar horas rechazadas.
\end{itemize}

\subsubsection{Archivos}
Ruta base: \texttt{/files}. Este conjunto de \textit{endpoints} gestiona la descarga o previsualización de los archivos
de la plataforma. Esto incluye fotos de perfil, recursos de proyectos y evidencias de horas.
\begin{itemize}
    \item \texttt{GET /files/:id/download}: Descargar archivo.
    \item \texttt{GET /files/:id/preview}: Previsualizar archivo.
\end{itemize}

\subsubsection{Otros}
Adicionalmente, se definieron otros \textit{endpoints} complementarios que principalmente permiten consultar datos
disponibles en la plataforma.
\begin{itemize}
    \item \texttt{GET /faculties}: Listar facultades disponibles.
    \item \texttt{GET /careers}: Listar carreras disponibles.
    \item \texttt{GET /campuses}: Listar campus disponibles.
    \item \texttt{GET /tags}: Listar etiquetas disponibles.
\end{itemize}

\section{Fase 2: Desarrollo del \textit{Backend}}
La segunda fase se centró en la implementación del \textit{Backend} de la plataforma, capaz de soportar y gestionar
todas las funcionalidades definidas en la fase anterior. 

El objetivo de esta fase es desarrollar una API estructurada capaz de gestionar proyectos de extensión por parte de
los usuarios. Esto incluye la gestión de usuarios, proyectos, postulaciones, horas y archivos.

\subsection{Selección de tecnologías}

\subsubsection{Entorno de ejecución y \textit{Framework}}
Se seleccionó \textbf{Node.js} como entorno de ejecución, utilizando el \textit{framework} \textbf{Express}.
Node.js ofrece una ejecución de código rápido, una de sus altas ventajas en un entorno orientado al
\textit{Backend} es que tiene un modelo I/O no bloqueante, lo cuál le permite manejar gran cantidad de solicitudes
simultáneas de manera eficiente. Gracias a esto, el entorno Node.js permite que la API opere de manera 
asíncrona y facilitando el manejo de solicitudes simultáneas sin afectar el rendimiento.
Express, por su parte, proporciona una estructura flexible y minimalista, lo cuál agiliza el desarrollo y facilita 
aspectos de mantenibilidad y escalabilidad.

\subsubsection{Compilador}
Para aumentar la robustez del código y tener un desarrollo más eficiente y seguro, se seleccionó \textbf{TypeScript}
para introducir tipado estático en el proyecto. Con el fin de evitar errores durante el desarrollo y mejorar la
legibilidad del código.

\subsubsection{Persistencia de datos}
Se optó por una base de datos relacional utilizando \textbf{PostgreSQL} como motor de base de datos para el
almacenamiento de los datos de la plataforma. Esto gracias a su robustez, escalabilidad, soporte a tipos de datos
avanzados (como UUID, gracias a la extensión pgcrypto).

Además, para el almacenamiento de archivos se optó por \textbf{Amazon S3}. Amazon Web Services (AWS) es un proveedor
ampliamente reconocido y confiable en la industria de servicios en la nube, provee una alta escalabilidad, seguridad,
y una sobresaliente disponibilidad, lo que lo convierte en el mejor candidato para el almacenamiento persistente de
archivos que la plataforma requiere. Además, AWS ofrece un SDK para JavaScript en Node.js, lo que facilita 
su integración con la API. Por lo tanto, el uso de \textit{Amazon S3} ofrecerá una alta disponibilidad, acceso seguro
a los archivos y un manejo de ciclos de vida de los mismos para garantizar la eficiencia y escalabilidad.

\subsection{Librerías}

\subsubsection{Seguridad de datos}
Con el fin de garantizar seguridad en los datos en reposo, se integró \textbf{bcryptjs} para un \textit{\gls{hashing}} seguro de
contraseñas.

\subsubsection{Autenticación y autorización}
Se integró \textbf{jsonwebtoken} para la gestión de tokens de autenticación, basando el manejo de sesiones con JWT. Facilitando 
la autorización basada en roles para los distintos \textit{endpoints} de la API.

\subsubsection{Validaciones}
Para la validación de datos de entrada se utilizó \textbf{Joi}. Ya que además de la definición de esquemas de validación robustos
y reutilizables, tiene una integración sencilla como \textit{middleware} para las peticiones recibidas del lado del servidor.

\subsubsection{Envío de correos electrónicos}
Para el envío de correos electrónicos se utilizó \textbf{Nodemailer}. Esto para el envío de correos como el de verificación de 
correo electrónico, restablecimiento de contraseña, entre otros.

\subsubsection{Manejo de archivos}
Para el manejo de archivos se integraron dos librerías importantes, la primera \textbf{Multer} para el procesamiento de archivos
en las solicitudes entrantes que permite realizar validaciones y manejo de los archivos, y la segunda el SDK (v3) de AWS para 
que permite realizar operaciones de carga, consulta y eliminación de archivos en S3.

\subsubsection{Monitoreo}
Para el monitoreo del sistema se utilizó \textbf{Winston} con el fin de generar \textit{\gls{log}} de eventos relevantes del sistema,
como errores, acciones de usuarios, entre otros.

\subsection{Arquitectura}
Para mantener una organización y separación de responsabilidades clara, el proyecto utiliza una arquitectura por capas.

\subsubsection{Estructura del proyecto}
Ya que se seleccionó una arquitectura por capas, se estructuró el proyecto en capas lógicas, cada una con responsabilidades específicas:
\begin{itemize}
    \item \textbf{Routers:} Esta capa se encarga de definir y exponer los \textit{endpoints} con el fin de interceptar las
            solicitudes entrantes y dirigirlas al controlador correspondiente.
    \item \textbf{Middlewares:} Esta capa (intermedia) se encarga de procesar las solicitudes entrantes y permite realizar
            validaciones como autenticación, autorización, validación de datos, manejo de errores, entre otros.
    \item \textbf{Controllers:} Esta capa se encarga de recibir las solicitudes enviadas por los routers, procesar la información
            de entrada e invocar los servicios correspondientes y devolver una respuesta al cliente.
    \item \textbf{Services:} Esta capa se encarga de toda la lógica de negocio, orquestando operaciones, reglas de negocio y permitiendo
            reutilización de servicios.
    \item \textbf{Repositories:} Esta capa se encarga de la interacción directa con la base de datos, realizando operaciones CRUD de
            las entidades y consultas directas.
\end{itemize}

Esta estructura fue aplicada a cada uno de los módulos definidos en la fase anterior.

\subsubsection{Archivos adicionales}
Además de las capas mencionadas, se crearon directorios y archivos adicionales para configuración de la plataforma, así como archivos
de conexión (a la base de datos, a AWS S3 y servicios de correo electrónico), entre otros.
\begin{itemize}
    \item \textbf{Types:} Este directorio contiene las definiciones de tipos utilizados en el proyecto.
    \item \textbf{Validators:} Este directorio contiene esquemas de validación de datos de entrada utilizando Joi.
    \item \textbf{DB:} Este directorio contiene el \textit{\gls{script}} para la creación completa de la base de datos.
    \item \textbf{Config:} Este directorio contiene archivos de configuración global.
    \item \textbf{Connections:} Archivos de conexión a la base de datos, AWS S3 y servicios de correo electrónico.
\end{itemize}

\subsection{Desarrollo de la API}
Una vez modularizado y estructurado el proyecto, se procedió a desarrollar los \textit{endpoints} definidos en la fase anterior para
cada uno de los módulos. Bajo una constante iteración de desarrollo, pruebas, corrección de errores y refactorizaciones para garantizar
un código limpio, mantenible y bajo la premisa \gls{dry} se logró desarrollar una API funcional que cumple con los requerimientos
definidos para cada módulo. Cada uno de los módulos protegidos con \textit{rate limiting} y \textit{tokens} cuando aplica.

\subsubsection{Autenticación y autorización}
Para el módulo de Autenticación se implementó una autenticación basada en \textit{Access Tokens} (corta duración, 1 hora)
y \textit{Refresh Tokens}
(larga duración, 7 días), con rotación de \textit{tokens} cuando un usuario renueva su sesión (\textit{refresh token}).
El \textit{Access Token}, generado con JWT, incluye información relevante del usuario así como su rol, lo que permite una autorización
basada en roles para los distintos \textit{endpoints} de los demás módulos. El \textit{Refresh Token} se almacena de manera segura
en la base de datos, asociado al usuario y a la sesión específica, lo que permite gestionar múltiples sesiones por usuario y
revocar sesiones de manera individual. Este último se envía al cliente como una \textit{cookie} HTTP-only, con el fin aumentar la seguridad.
Todos los usuarios deben de verificar su correo electrónico por medio de un \textit{token} de verificación enviado por medio de correo electrónico.

\subsubsection{Gestión de usuarios}
Para el módulo de Usuarios se implementó la gestión básica de usuarios. Esto incluye la consulta, actualización y eliminación de usuarios.
Se crearon reglas de negocio para la actualización y eliminación de usuarios, como que un usuario pueda actualizar solamente su propia información. Así
como temas de autenticación, como el cambio de contraseña.
Adicional, las organizaciones deberán de ser aprobadas por algún director para poder tener su cuenta habilitada.

\subsubsection{Ciclo de vida de proyectos de extensión}
Para el módulo de Proyectos, se implementó un ciclo de vida completo para los proyectos en los que se incluyen los estados: \textit{draft},
\textit{open}, \textit{in progress}, \textit{completed} y \textit{cancelled}. Una vez un Director u Organización externa crea un proyecto,
este se encuentra en estado \textit{draft} hasta que un Director lo aprueba, momento en el que el proyecto pasa a estado \textit{open} y los
estudiantes pueden ver y llegar a postularse. Una vez el dueño del proyecto inicia el proyecto, este pasa a estado \textit{in progress}
(ya no se aceptan postulaciones). Una vez el proyecto finaliza, el dueño del proyecto lo marca como finalizado, pasando a estado
\textit{completed}, permitiendo a los estudiantes que participaron en el proyecto registrar sus horas trabajadas. En caso de que el proyecto
sea cancelado, el dueño del proyecto lo marca como cancelado, pasando a estado \textit{cancelled}, lo que permite a los estudiantes indicar
las horas trabajadas, si aplica.

\subsubsection{Ciclo de vida de partipación en proyectos de extensión}
Para el módulo de Postulaciones, se implementó un ciclo de vida completo para estas, en el que se incluyen los estados: \textit{pending},
\textit{accepted}, \textit{rejected} y \textit{finalized}. Al momento en el que un estudiante se postula para participar en un proyecto,
la postulación se encuentra en estado \textit{pending} hasta que el dueño del proyecto la aprueba o rechaza, momento en el que la
postulación pasa a estado \textit{accepted} o \textit{rejected}, respectivamente. En caso de que la postulación sea aprobada, el estudiante
se convierte en participante del proyecto y no podrá postularse o participar en otros proyectos hasta que su participación actual concluya.
Una vez el proyecto finalice o el estudiante abandone el proyecto, la postulación pasa a estado \textit{finalized}.

\subsubsection{Ciclo de vida de horas trabajadas en proyectos de extensión}
Para el módulo de Horas, se implementó un ciclo de vida completo para estas, en el que se incluyen los estados:  \textit{pending},
\textit{approved}, \textit{rejected}. Al momento en el que un estudiante registra horas trabajadas en un proyecto, estas se encuentran
en estado \textit{pending} hasta que el dueño del proyecto las aprueba o rechaza, momento en el que las horas pasan a estado
\textit{approved} o \textit{rejected}, respectivamente. En caso de que las horas sean rechazadas, el estudiante tendrá la opción de corregir
la cantidad de horas indicadas y reenviar las horas para su aprobación, este proceso se podrá realizar un máximo de 3 veces, después de
lo cual las horas quedarán en estado \textit{rejected} de manera permanente.

\subsubsection{Manejo de archivos}
Para el módulo de Archivos, se utilizó el SDK de AWS S3 para poder realizar operaciones de carga, consulta y eliminación de archivos.
Dentro de la instancia de S3 se creó un \textit{bucket} con cuatro directorios: \textit{profilepic} para almacenar las fotografías de perfil
de los usuarios, \textit{resource} para almacenar los archivos relacionados a los proyectos, \textit{evidence} para almacenar los
archivos relacionados al reporte de horas trabajadas por los estudiantes y \textit{projectImage} para almacenar las imágenes colocadas
en los editores de texto enriquesidos, los cuales forman parte de la descripción, justificación y requerimientos de un proyecto.
Los directorios de \textit{resource} y \textit{evidence} se organizan internamente con subdirectorios por proyecto o usuario; 
además, estos directorios son de acceso privado. Por lo que para consultar o descargar un archivo, se genera un 
\textit{pre-signed URL} con una duración limitada, que permite acceder al archivo de manera segura sin exponer el acceso 
directo a S3. Por otro lado, el directorio de \textit{profilepic} y \textit{projectImage} son de acceso público, esto debido a que
estos archivos necesitan ser accedidos de manera frecuente para la visualización de perfiles y proyectos, respectivamente, y su contenido
es de acceso público por naturaleza.
Cada vez que se realiza la carga de un archivo, el primer paso es validar que el archivo cumpla reglas de tamaño 
máximo, extensiones, entre otras validaciones. Luego, el archivo se carga a S3 en su respectivo directorio, esta acción retorna 
un nombre único conocido como \textit{key}. Esta \textit{key}, junto con otra información relevante, se almacena en la base de 
datos con sus respectivas relaciones con usuarios, proyectos u horas, dependiendo del caso.

\section{Fase 3: Implementación del \textit{Frontend}}

La tercera fase se centró en la implementación del \textit{Frontend} de la plataforma, utilizando los prototipos de interfaz de
usuario previamente diseñados por Andrés Rodríguez en \cite{Rodríguez_Dis_2023} como guía ya validada para la construcción de la
interfaz de usuario.

El objetivo de esta fase es desarrollar la interfaz de usuario que permita a los usuarios interactuar con la plataforma web,
utilizando la API desarrollada en la fase anterior para gestionar usuarios, proyectos de extensión, postulaciones, horas y archivos.

\subsection{Selección de tecnologías}

\subsubsection{Entorno de construcción y librería principal}
Se seleccionó \textbf{Vite} como entorno de construcción, utilizando la librería \textbf{React}. Vite ofrece un servidor de
desarrollo rápido, optimizado y con configuraciones ideales para un proyecto moderno de \textit{Frontend}. Gracias a su
\gls{hmr} permite un desarrollo eficiente y fluido. React, por su parte, es una librería que facilita la construcción de
componentes reutilizables, gestionando el estado de la aplicación de manera eficiente y permitiendo basar una arquitectura
de reutilizable y mantenible.

\subsubsection{Compilador}
Al igual que en el \textit{Backend}, se seleccionó \textbf{TypeScript} para introducir tipado estático. Aumentando la
robustez del código, evitando errores durante el desarrollo y mejorando la legibilidad del código. Además, TypeScript
se integra de manera nativa con React, lo que facilita su uso en el proyecto. Adicional a esto, se integró \textbf{ESLint}
para mantener un estilo de código consistente y evitar errores comunes, así como \textbf{Prettier} para el formateo
automático del código.

\subsection{Librerías}

\subsubsection{Interfaz de usuario y estilos}
Con el fin de garantizar una interfaz de usuario alineada con los diseños de interfaz y con una línea gráfica coherente,
se optó por utilizar \textbf{Material UI}, ya que además de que ofrece una amplia variedad de componentes preconstruidos
y personalizables para adaptar el diseño y colores para alinear la identidad visual de la plataforma, tiene integración
nativa para TypeScript.

\subsubsection{Enrutamiento}
Para la gestión de rutas y navegación dentro de la plataforma se seleccionó \textbf{TanStack Router}, ya que ofrece
enrutamiento tipado, gestión de datos asíncronas (\textit{preloaders}) y rutas anidadas, entre muchas otras
características. Su enrutamiento basado en archivos permite una organización y entendimiento del proyecto más sencillo.

\subsubsection{Consumo de la API}
Para la gestión de datos asíncrona y comunicación con la API se seleccionó \textbf{TanStack Query}, ya que ofrece
sincronización automática para los datos, manejo de estados durante el proceso de \textit{fetching}
(cargando, error, éxito, entre otros), almacenamiento en caché para optimizar el rendimiento. Adicionalmente, se integró
\textbf{Axios} como cliente HTTP para realizar las solicitudes a la API, gracias a su manejo de promesas, interceptores.
TanStack Query admite cualquier cliente HTTP, por lo que Axios se integra de manera sencilla.

\subsubsection{Manejo de estados}
Finalmente, para el manejo de estados globales de la plataforma se seleccionó \textbf{Zustand}, ya que es ligero y
optimizado para el rendimiento, permitiendo una gestión de estados simple pero efectiva. Para la validación de
formularios se seleccionó \textbf{Zod} gracias a su enfoque en TypeScript y la construcción de esquemas de validación
robusta y reutilizable. Para el manejo de estados de formularios se seleccionó \textbf{TanStack Form} gracias a su
soporte para múltiples \textit{frameworks} y librerías, entre ellas, React, así como su integración con distintas
librerías de validación como Zod.

\subsection{Arquitectura}
La arquitectura seleccionada fue la arquitectura \textit{Single Page Application} (SPA, por sus siglas en inglés)
utilizando la estrategia de renderizado web \gls{csr}. Esto debido a que debido a la naturaleza de la plataforma,
se requiere que el usuario pueda interactuar con la aplicación de manera fluida, delegando la carga del renderizado
al cliente para evitar recargas completas de la página.

\subsubsection{Estructura del proyecto}
Con el fin de mantener un código modular, organizado, mantenible y escalable, se estructuró el proyecto en directorios
siguiendo \textit{feature-based} (basada en características), por lo que se separaron las responsabilidades.
Resultando en los siguientes directorios principales:
\begin{itemize}
    \item \textbf{Routers:} Este directorio contiene la configuración de las rutas de la aplicación empleando TanStack
            Router. Definiendo el árbol de rutas para tener el mapa de navegación de la aplicación.
    \item \textbf{Pages:} Este directorio contiene los componentes de página, que representan las distintas vistas o
            pantallas de la aplicación.
    \item \textbf{Components:} Este directorio contiene los componentes reutilizables de la aplicación.
    \item \textbf{Services:} Este directorio contiene los servicios de la aplicación, específicamente, los servicios de API.
    \item \textbf{Stores:} Este directorio contiene los archivos relacionados con el manejo de estados globales.
\end{itemize}

\subsubsection{Archivos adicionales}
\begin{itemize}
    \item \textbf{Types:} Este directorio contiene las definiciones de tipos utilizados en el proyecto.
    \item \textbf{Styles:} Este directorio contiene un único archivo que configura el \textit{theme} de Material UI
            para toda la aplicación.
    \item \textbf{Utils:} Este directorio contiene utilidades y configuraciones utilizadas en la aplicación.
    \item \textbf{Schemas:} Este directorio contiene los esquemas de validación de datos.
    \item \textbf{Context:} Este directorio contiene los contextos de React utilizados en la aplicación.
\end{itemize}

\subsection{Desarrollo de la interfaz de usuario}
Una vez definida la estrategia de renderizado, arquitectura y estructura del proyecto se procedió a desarrollar la
interfaz de usuario utilizando los prototipos de interfaz de usuario previamente diseñados por Andrés Rodríguez en
\cite{Rodríguez_Dis_2023}. Ya que dichos diseños fueron validados tanto por usuarios finales como por expertos en el área.

Luego de varias iteraciones, pruebas, correcciones y refactorizaciones, se logró desarrollar una interfaz de usuario funcional,
\textit{responsive} y alineada con los diseños de interfaz, que permite a los usuarios interactuar con la plataforma de una manera
fuida y eficiente. Utilizando correctamente la API desarrollada en la fase anterior para lograr gestión de usuarios, proyectos de
extensión, postulaciones, horas y archivos por parte de los usuarios desde la plataforma.

\subsubsection{Pantallas}
Las pantallas, previamente diseñadas, necesarias para la plataforma son las siguientes:
\begin{itemize}
    \item \textbf{Home:} Esta pantalla es la página de inicio de la plataforma. Tiene dos variantes, una para usuarios no autenticados
            y otra para usuarios autenticados. Sus elementos dependen del rol del usuario.
    \item \textbf{Registro:} Pantalla que permite a los nuevos usuarios crear una cuenta en la plataforma. Incluye un formulario de
            registro con campos específicos para cada tipo de usuario.
    \item \textbf{Inicio de sesión:} Pantalla que permite a los usuarios existentes iniciar sesión en la plataforma.
    \item \textbf{Verificación de correo electrónico:} Pantalla que permite a los usuarios verificar su dirección de correo
            electrónico después de registrarse.
    \item \textbf{Restablecimiento de contraseña:} Pantalla que permite a los usuarios solicitar un restablecimiento de contraseña.
    \item \textbf{Perfil:} Pantalla que muestra la información de un usuario.
    \item \textbf{Motivaciones:} Pantalla que muestra las motivaciones disponibles para los usuarios.
    \item \textbf{Edición de perfil:} Pantalla que permite a los usuarios editar su información de perfil.
    \item \textbf{Cambio de contraseña:} Pantalla que permite a los usuarios cambiar su contraseña.
    \item \textbf{Proyectos propios:} Pantalla que muestra los proyectos creados por el usuario. Visible para directores y organizaciones externas.
    \item \textbf{Creación de proyecto:} Pantalla que permite a directores y organizaciones externas crear un nuevo proyecto en la plataforma.
    \item \textbf{Edición de proyecto:} Pantalla que permite a directores y organizaciones externas editar un proyecto existente.
    \item \textbf{Búsqueda de proyectos:} Pantalla que permite a los usuarios buscar proyectos en la plataforma. Los filtros de búsqueda
            dependen del rol del usuario.
    \item \textbf{Visualización de proyecto:} Pantalla que permite a los usuarios visualizar los detalles de un proyecto. Los elementos
            visibles dependen del rol del usuario.
    \item \textbf{Configuración de proyecto:} Pantalla que permite a directores y organizaciones externas cambiar el estado de un proyecto,
            modificar los participantes o aceptar/rechazar postulaciones.
    \item \textbf{Proyecto actual:} Pantalla que muestra la información del proyecto en el que se está participando actualmente. Visible para estudiantes.
    \item \textbf{Historial de proyectos}: Pantalla que muestra el historial de proyectos en los que se ha participado. Visible para estudiantes.
    \item \textbf{Registro de horas:} Pantalla que permite a los estudiantes registrar las horas trabajadas en un proyecto, incluyendo la carga de evidencias.
    \item \textbf{Horas pendientes de aprobación:} Pantalla que muestra las horas registradas por los estudiantes que están en espera de ser aprobadas. Visible para directores y organizaciones externas.
    \item \textbf{Proyectos pendientes de aprobación:} Pantalla que muestra los proyectos que están en espera de ser aprobados. Visible para directores.
    \item \textbf{Organizaciones externas pendientes de aprobación:} Pantalla que muestra las organizaciones externas que están en espera de
            ser aprobadas. Visible para directores.
\end{itemize}

\subsubsection{Definición de tema}
Utilizando la característica de personalización de temas de Material UI, se definieron los siguientes aspectos para el tema global de la aplicación:
\begin{itemize}
    \item \textbf{Paleta de colores:} Se definieron colores primarios, secundarios, de fondo, de texto, error, información, éxito, entre otros.
    \item \textbf{Tipografía:} Se definieron las fuentes (Roboto Mono y Raleway), tamaños, pesos y estilos.
    \item \textbf{\textit{Breakpoints}:} Se definieron los puntos de quiebre para \textit{desktop}, \textit{tablet} y \textit{mobile}
            para el diseño \textit{responsive}.
\end{itemize}

\subsubsection{Rutas y navegación}
Debido a cómo TanStack Router maneja las rutas, dentro del directorio de Routers se definieron los subdirectorios y archivos necesarios
para cubrir todas las pantallas mencionadas anteriormente. Todos los archivos de rutas se encargan de definir la ruta, el componente a
renderizar y parámetros si aplica.

\subsubsection{Componentes}
Para la construcción de componentes se utilizó principalmente los componentes ofrecidos por Material UI. Cada uno de los componentes
fueron ordenados en subdirectorios dentro del directorio de \textit{Components} con el fin de mantener una organización clara.

\subsubsection{Formularios y validaciones}
Para manejar el estado de los formularios, TanStack Form ofrece un hook llamado \texttt{useAppForm}, el cuál permite el uso de componentes previamente
implementados como parte de un formulario. Logrando una fácil gestión del estado del formulario en conjunto con una integración de Zod para validación
síncrona de los datos de entrada.

\subsubsection{Integración de la API}
Para la integración de la API se utilizó TansStack Query y Axios. TanStack Query se encarga del ciclo de vida de las peticiones, mientras que Axios 
se encarga de realizar dichas peticiones a la API. Axios fue configurado con dos intercepotores, uno para incluir el token de autenticación en las
solicitudes y otro para manejar errores globales. Una configuración importante fue habilitar el uso de \textit{cookies} en las solicitudes para permitir 
la renovación de tokens de autenticación. 

\subsubsection{Consumo de la API}
Cada uno de los módulos de la API fue integrado al \textit{Frontend} bajo dos formas. Para cada módulo se crearon:
\begin{itemize}
    \item \textbf{Servicios:} Contienen funciones puras (las llamadas realizadas con Axios), \textit{hooks} de TanStack Query
            (envuelven las funciones puras) e invalidaciones (para forzar resincronizaciones) y utilidades relacionadas a cada módulo.
    \item \textbf{\textit{Hooks} personalizados:} Contienen y llaman a los servicios, por lo que contienen los datos retornados por Axios
            y los estados de las peticiones entregados por TanStack Query. Estos \textit{hooks} son los que se utilizan dentro de los
            componentes para consumir la API.
\end{itemize}

\subsubsection{Gestión de estado global}
El único estado global que se maneja en la plataforma es la autenticación del usuario. Esto encapsula información relevante de una
sesión, como si el usuario se encuentra autenticado, rol, nombre, foto de perfil, entre otros. Ligado a esto, se crearon dos
\textit{hooks}. El primero para la información de autenticación del usuario, y el segundo para la validación de permisos de acceso
a rutas o elementos basados en el rol del usuario. 

\section{Fase 4: Pruebas de funcionalidad y usabilidad}
La cuarta fase se centró en la realización de pruebas de funcionalidad para el \textit{Backend} y pruebas de usabilidad 
para el \textit{Frontend}.

El objetivo de esta fase es garantizar que la plataforma web es capaz de realizar todas las funcionalidades definidas en la fase 1
para cada módulo, así como garantizar que la interfaz de usuario ofrece el acceso a dichas funcionalidades de manera correcta
y alineada con los diseños de interfaz utilizada.

\subsubsection{Backend}
Para el \textit{Backend} se busca garantizar que la API es capaz de brindar todas las funcionalidades definidas en la fase 1 para cada
módulo. Esto incluye que cada uno de los \textit{endpoints} definidos en la fase 1 funcione correctamente, aceptando y retornando los
datos esperados, manejo correcto de la autorización basada en roles, así como el manejo de errores.
